End-to-End Demo Flow

MLE 自动触发 feature 分析检测

从某个模型的 performance、残差、segment loss 和业务目标出发，结合宽表 schema 找潜在有价值的列，再对候选列做 profile。

  • 分布特性：p25/p50/p75/p90/p99、std、variance、skew、outlier rate。
  • 数据质量：coverage、null rate、zero rate、freshness、sample stability。
  • 建模价值：label correlation、mutual information、segment lift、SHAP/feature importance proxy。
  • 可生产性：cardinality、dimension family cost、point-in-time leakage risk、join rate。

Dashboard UI 需要呈现完整迭代过程

UI 不只是字段浏览器，还需要展示宽表内容、aggregation registry、候选 feature 的公式、下发命令、pipeline 运行、GMinor join 和 ML 复测结果。

  • 宽表全量字段、source 前缀、schema version、profiling status。
  • aggregation 内容：dimension family、window、measure、公式和数学运算。
  • 迭代细节：发现原因、候选列、profile 证据、命令状态、训练结果和下一轮建议。

四个模块的大概内容

1

WideTable

定义实时归因宽表契约，并把 Jaeger / Scrat shadow path 落到可编码 spec。

  • 事件级 source of truth，不提前聚合。
  • 保留 event_id、imp_id、transaction ids。
  • HB POC: Jaeger 写 KVDB，Scrat lookup 后写 wide topic。
  • 强调 async、bounded、best-effort，不能影响 serving latency。
2

Data

先用离线 batch join 生产 golden wide dataset，作为实时 POC 的评估基线。

  • 读取 coba2.ex_jaeger_transaction 和 coba2.hb_transactions。
  • 三个 Spark jobs: Jaeger staging、HB staging、wide join。
  • schema 驱动生成 DDL、columns、col maps 和 SQL templates。
  • 默认按 event-id hash sample 控制成本。
3

MachineLearning

把 wide-table-derived features 接入离线训练、误差分析和候选 feature 验证。

  • 构建 point-in-time training / eval dataset。
  • 比较 GMinor baseline prediction 与 treatment model。
  • Round 1 DCNv2 HP search 已证明当前瓶颈更像 feature gap。
  • Round 2 聚焦 price volatility 和 format-specific signals。
4

Dashboard

Feature Workbench，把 schema、能力、聚合配置、公式和 simulation 结果可视化。

  • Capability Map 展示字段覆盖、freshness、null rate。
  • Aggregation Builder 限制 dimension family 和成本。
  • Formula Studio 做类型检查和 leakage 检查。
  • Simulation Lab 对比 baseline / treatment、lift 和 cohort diagnostics。

通过 schema 切分成员分工

协作方式采用 schema-first：先把 wide event schema、字段前缀、semantic type、null semantics、feature suitability、 allowed dimension families 和 leakage rules 固定下来，再由不同成员按字段族并行实现 ingestion、codegen、feature capability 和 UI。

1

Schema contract

定义事件粒度、join keys、字段前缀和每列语义，避免 Jaeger/HBN/ML/UI 各自解释字段。

2

Generated artifacts

从 schema 生成 columns JSON、col maps、DDL 和 SQL templates，Data jobs 和 Dashboard 共用同一份元数据。

3

Prefix ownership

成员按字段族负责：identity / attribution、jgr_*、hbn_*、feature metadata、simulation schema。

4

Review by compatibility

review 重点看字段兼容性、点时间正确性、成本边界和是否改变已有消息语义。

问题和痛点

event_id 不一定跨源稳定

如果 HBN/TPAT 和 delivery 使用不同 id，即使 KVDB TTL 足够也会 miss。需要 Phase 0 key coverage 和 fallback key normalization。

跨 Kubernetes cluster 归因

delivery 可能在 cluster A，notification 到 cluster B。测试单位必须是 global event-id sampled cohort，而不是某个 cluster。

7-day hot KV 成本过高

全量 7 天 hot KV 容易到 TB/PB 级。demo 应先用短 TTL、compact projection、cold log repair 和 measured delay CDF 决策。

serving latency 不能受影响

Jaeger writer 必须 async、bounded、non-blocking，KVDB outage 只能表现为 metrics/logs，不能影响 auction response。

feature leakage 和 point-in-time correctness

Feature formula 不能引用未来 label，同一 event 不应贡献给自己的 trailing feature，GMinor join 必须按 request_time 做点时间约束。

维度自由组合会失控

Dashboard 不能开放任意 dimensions 或 SQL。必须使用 reviewed dimension families、sample-first execution 和 row-count/cost gates。

schema drift 会破坏 downstream

字段含义、null semantics、前缀规则和 generated artifacts 必须从同一 schema source 派生，并在 CI 中做 validation。

offline 指标不等价于 online A/B

Demo 能证明 candidate feature 值得 productionize，但完整 counterfactual 仍依赖 bid response array、served feature log 和 policy replay semantics。

数据质量和缺失原因要可解释

lookup miss 要拆成 sampled out、no key、expired、source key mismatch、write failure 等原因，否则无法判断架构是否有效。

UI 和全流程交互接口还需要设计

Dashboard、MLE analyzer、aggregation compiler、pipeline runner 和 simulation runner 之间需要明确 command schema、状态回传、权限边界和失败重试语义。

是否能真正探索出有价值的数据需要验证

当前假设是宽表能暴露新的有效信号，但仍需用离线实验确认候选列能稳定提升目标模型，而不是只解释历史误差或引入泄漏。

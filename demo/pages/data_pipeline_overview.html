The pipeline — coba2 landings → wide table

Explode Jaeger placement results to one row per (ad_event_id, imp_id), then LEFT JOIN the served/winning HB bid. The same event-id hash-sample gate runs on both sides so they cover one event-id cohort. Codegen ✓ artifacts consistent with schema (validate.py exit 0).

Why hbn_* null is a signal, not missing data

The wide table is built from the served/attempted impression grain; the HB bid is joined LEFT. A null hbn_* means "not HB/S2S" — a meaningful category, not a data gap. Overlapping concepts are deduped to a single authoritative source; the dropped copy is not carried as a column.

Binding constraint is write throughput, not capacity

~1–2.3 GB/s logical writes at 330–415K SET/s, amplified ~10–30× by leveled compaction ⇒ ~10–70 GB/s physical disk. Biggest lever: store a projection (join/debug fields, ~hundreds of bytes) instead of the full ~3 KB message → 5–10× reduction.

Pain points — where the loop still needs a human

The pipeline is schema-driven and CI-linted, but a few steps still break the autonomous loop. Below: the friction, and the fix we'd propose.

Manual steps
1 Build, rerun & promotion are hand-driven manual · every code change + handoff
Pain
  • JAR build loop — every Scala edit means push → wait for the GitHub Action to build the assembly JAR → then manually resubmit the SparkApplication. The CI round-trip is the slow part of each iteration, and a human has to babysit the trigger and the rerun; the agent can't close the loop on its own.
  • Promotion into lena — moving the jobs into the lena tree means manually relocating src/main/scala + src/main/resources, switching the package base to com.vungle.lena.<domain>, and confirming boilerplate helper names before sbt compile — an error-prone, one-off rename.
Proposed fix
  • Close the build loop — grant a GitHub token so the agent triggers the workflow, polls it to green, and resubmits hands-off; or build the JAR locally (sbt assembly) and push straight to the S3 JAR path the SparkApplication reads, skipping CI for dev iterations.
  • Script the promotion — a single step (package rewrite + path move + helper-name check) so the lena handoff is one command, not a manual port.
2 Fields & enums that can't be auto-synced manual · per schema / enum change
Pain
  • Fields that can't be auto-mapped — the codegen maps each schema field to a coba2 source leaf, but can't when the source won't resolve. validate.py already flags serve_result.pd_cpx, ad_podding_multiplier, double_verify_fraud_reason, bidrequest_time as source leaf not in the coba YAMLs. Two modes: name mismatch (field name ≠ upstream leaf, no exact match) and upstream null / absent (the leaf isn't emitted or is all-null, so there's nothing to map).
  • Enum source-of-truth drift — enum_ref codes live in schema §9 but their real source is upstream EnumeratedList.md (jaeger / hb-notifications). When upstream adds a code, the schema doesn't know — someone has to notice and hand-sync.
Proposed fix
  • Fuzzy alias + nullability flag — agent proposes a best-guess alias for near-misses, human confirms once, and the pin lands in the col_map so it's stable thereafter; a per-field expected-nullability flag (an hbn_* null is signal; a jgr_* null usually isn't) so validation only escalates the real gaps.
  • Scheduled enum diff — diff the upstream EnumeratedList.md against schema §9 and open a PR (or flag the agent) when they diverge — same pattern as the existing kafka-enum sync.
